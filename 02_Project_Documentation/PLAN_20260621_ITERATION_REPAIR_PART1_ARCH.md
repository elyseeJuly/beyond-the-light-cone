# 迭代修复计划 - Part 1：底层架构与系统基建重构
> **Date**: 2026-06-21  
> **Status**: Planned  
> **Category**: Iteration Plan (`PLAN_`)

本阶段重点解决已发现的严重代码架构风险（P0/P1），通过彻底解耦和健壮性建设，为后续业务逻辑提供稳定支撑。

## 1. 风险点响应矩阵

| 风险等级 | 问题描述 | 核心诉求 | 计划模块 |
|:---|:---|:---|:---|
| **P0** | `Game.ts` 巨型上帝类与伪 DI 容器，高度耦合，调度逻辑过重。 | 拆分调度逻辑，实现真正的依赖注入（DI）。 | 架构解耦模块 |
| **P1** | 存档系统脆弱：整棵对象图序列化，缺乏 Schema 及双写一致性保障。 | 建立版本迁移路径，精简存档结构。 | 存档重构模块 |
| **P1** | 测试覆盖率存在“虚假安全感”：依赖全局 Mock 的重度集成测试。 | 补充纯粹的单元测试，覆盖基础模块。 | 测试体系革新模块 |

## 2. 详细实施方案

### 2.1 架构解耦：Game.ts 重构与真·依赖注入 (P0)

**现状问题**：
`Game.runARound` 方法承担了数值结算、事件触发、UI更新等所有逻辑。且各模块仍直接调用 `GameInstance.get()` 获取单例，违背了控制反转（IoC）原则，导致后续极高的合并冲突率。

**修复计划**：
1. **引入标准 DI 容器**：使用类似 `InversifyJS` 的轻量级容器，或手写标准的 IoC 注册机制。
2. **拆分 God Class**：
   - 将 `Game.ts` 拆分为 `GameLoopCoordinator`（主循环协调器）。
   - 将具体业务逻辑下放到对应的子系统：`EconomySystem`（经济）、`EventSystem`（事件）、`PopulationSystem`（人口/角色）。
3. **消除全局单例强耦合**：
   - 彻底废弃并移除 `GameInstance.get()`。
   - 所有子系统和服务通过构造函数或属性注入的方式获取依赖（例如 `constructor(private eventSystem: IEventSystem)`）。

### 2.2 存档系统重构：Schema 验证与迁移 (P1)

**现状问题**：
整棵内存对象图直接 `JSON.stringify`，新增字段导致旧存档崩溃；同时 LocalStorage 与 IndexedDB 双写缺乏事务保障。

**修复计划**：
1. **定义 Save Schema (存档契约)**：
   - 提取需要持久化的最小状态树（如 `resources`, `unlockedFlags`, `currentYear`），而非整个类实例。
   - 引入 Zod 等 Schema 校验库，对读取的存档数据进行严格格式校验。
2. **实现存档版本迁移 (Migration)**：
   - 为存档增加 `version` 字段。
   - 建立 `MigrationRegistry`，编写从旧版本 `v1 -> v2`, `v2 -> v3` 的升级脚本，确保兼容老玩家数据。
3. **统一存储策略**：
   - 废除 LocalStorage 和 IndexedDB 的直接双写策略。
   - 采用单一数据源（推荐优先 IndexedDB），LocalStorage 仅作为 `save_index`（存档目录元数据）的备份。

### 2.3 测试体系革新：真实的单元测试 (P1)

**现状问题**：
800+ 用例实质为集成测试，过度 Mock 全局 Game 单例，一旦底层修改，所有测试失效。

**修复计划**：
1. **测试分层策略**：
   - **纯单元测试 (Unit Tests)**：剥离外部依赖，测试纯函数与独立模块（如：事件条件校验逻辑、资源增减计算器）。
   - **集成测试 (Integration Tests)**：仅保留核心流程（如一轮 GameLoop 运转）的少量集成测试。
2. **重构现有用例**：
   - 识别并重写那些“仅为了满足覆盖率而写”的脆弱测试。
   - 针对 DI 重构后的各个 `System` 编写孤立测试，传入真实的假对象（Fake）而非复杂的 Mock 链。
