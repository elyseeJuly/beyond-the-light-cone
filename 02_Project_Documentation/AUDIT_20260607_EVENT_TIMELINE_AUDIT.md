## 二、 纪元时间线与固定剧情事件审计 (Fixed Events Alignment)

目前 `events.json` 中的固定剧情事件（Milestone）通过绝对年份匹配触发。例如，年满 200 年强制触发威慑纪元事件。这会导致以下问题：
1. 若玩家的文化数值增长极快，在第 50 年就将纪元推向了“威慑纪元”，但新纪元的重要人物（程心、维德）必须等到第 200 年才会解锁。
2. 若玩家的文化增长极慢，在第 200 年仍处于“危机纪元”，事件却强行提示“威慑纪元到来了”。

### 重构方案：时序条件拦截
1. 修改 `GameEventManager.ts` 中的 `checkEvents()` 触发逻辑：
   - 触发时机由 `e.inYear === currentYear` 更改为 `currentYear >= e.inYear`。
   - 引入 `triggerCondition` 拦截机制。如果固定事件配置了 `triggerCondition`，必须在满足条件（例如 `epoch: "DETERRENCE"`）时才允许触发。若未满足，则该事件处于挂起状态，直到条件在未来某回合满足时才触发。
2. 在 `events.json` 中，为需要跟特定纪元同步的固定事件增加 `triggerCondition.epoch` 限制：
   - **威慑交接** (`inYear: 200`)：增加 `{ "epoch": "DETERRENCE" }` 约束。
   - **云天明童话** (`inYear: 260`)：增加 `{ "epoch": "BROADCAST" }` 约束。

## 三、 数值与事件的双向影响机制 (Bidirectional Feedback Loops)

以前的机制只支持**“事件影响数值”**，即玩家选择某个选项后，扣减/增加经济、文化、军事等。无法实现**“数值决定事件”**（即当军事或社会反叛值达到特定阈值时触发特殊的历史抉择或惩罚事件）。

### 重构方案：随机事件条件解析器
1. 升级 `GameEventManager.ts` 的 `checkRandomEvents()` 方法，在挑选候选事件时，将原来只检查 `epoch`、`reqTech`、`reqStar` 的简易判断，改用全局条件解析器 `checkFilterConditions()`（仅在匹配前剥离 `probability` 权重，防止权重双重衰减）。
2. 使随机事件（`randomevents.json`）可以支持以下条件拦截：
   - `minEconomy`、`minCulture`、`minPopulation`、`minDeterrence`：判定文明建设门槛。
   - `maxTreachery`：判定社会反叛或逃亡主义上限。
   - `reqFlag`、`reqNotFlag`：判定是否拥有某些决策因果标记。

### 触发示范
- **社会动荡事件**：当 `treachery >= 50` 时，解锁 `escape_panic_rebellion` 逃亡恐慌暴动随机事件，强制逼迫玩家在镇压与安抚间做出抉择，影响最终是否达成“文明崩溃”失败结局。
- **科技突破门槛**：某些高级研究相关的随机事件（如“重核聚变点火成功”）要求玩家 `minCulture >= 100` 或 `minEconomy >= 200` 才会刷出，反过来影响技术的非线性增长。
