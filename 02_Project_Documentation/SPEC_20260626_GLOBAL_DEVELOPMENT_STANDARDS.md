> **版本号 (Version)**: V1.2.0  
> **生效日期 (Effective Date)**: 2026-06-26  
> **适用对象 (Target)**: 所有项目开发、AI 协同助手、独立工程师与版本归档  
> **基于版本**: [V1.1.0 (2026-06-16)](file:///Users/quantumrose/Documents/Emberois/emberois-dev-standards/specifications/SPEC_20260616_GLOBAL_DEVELOPMENT_STANDARDS.md)

## 📖 一、概述与核心目的

随着工程和协作体系的不断扩大（多项目并行、AI 助手深度参与），维护一套高度一致、清晰易读且利于检索的文档管理规范是确保项目生命周期顺畅流转的根本。

本规范的核心目的在于：
1. **杜绝文档混乱**：通过科学的命名和时间排序，使文档在文件管理器或编辑器中实现"自然归类"和"时间线追溯"。
2. **AI 无缝接手**：为接手开发的 AI 助手提供绝对明确的上下文环境、任务交付格式与行为准则。
3. **沉淀历史轨迹**：确保每个重大的审计、规格说明、测试和会话执行都有迹可循，杜绝"代码跑通，思路黑盒"的情况。
4. **[V1.2.0 新增] 控制文档膨胀**：通过提高文档触发门槛和引入测试作为交接物，防止文档数量无限增长。

## 🛠️ 二、"分类+日期"双重命名公式

所有项目的开发文档（通常位于 `docs/` 或 `02_Project_Documentation/` 等文件夹内）必须严格遵守以下命名公式：

$$\Large \texttt{[CATEGORY\_PREFIX]\_[YYYYMMDD]\_[DOCUMENT\_NAME].md}$$

### 1. 字段解析说明：
*   **`[CATEGORY_PREFIX]`**：全局分类大写前缀之一（参见第三节），用以规范文档属性。
*   **`[YYYYMMDD]`**：文档创建日期或重大修改日期，必须使用 8 位数字。
*   **`[DOCUMENT_NAME]`**：精炼、清晰的英文大写名称，单词之间使用下划线 `_` 隔开。如果是特定国家的补充指令，可在尾部加上 `_CN`（例如 `_DIRECTIVES_V2_CN.md`）。

### 2. 核心优势：
*   **物理天然归组**：按文件名排序时，所有文档会根据所属分类（如 `SPEC_`、`AUDIT_`、`DICT_`）自动成组聚集，无需建立繁多的嵌套子文件夹。
*   **时间线完美追溯**：在同类别前缀下，文档会按 `YYYYMMDD` 升序自动从早到晚排列，便于浏览功能演进历史。

## 🏷️ 三、全局分类前缀定义

### 3.1 原有六大前缀（保持不变）

| 大写前缀 | 英文分类 (Category) | 适用文档范围 | 编写要求与规范 |
| :--- | :--- | :--- | :--- |
| **`SPEC_`** | **Specifications** (规格说明与设计系统) | 核心架构设计、React/TS UI 指令书、美术需求说明书、迁移方案、全局标准文档。 | 必须包含清晰的模块边界、开发规范约束、数据结构接口（Interface）定义及 AI 执行规则。 |
| **`AUDIT_`** | **Audits** (审计报告与系统平衡) | 代码静态审计、叙事与交互心流评估、难度/数值曲线平衡建议、性能优化案。 | 必须列出具体问题的描述、在代码中的具体证据文件/行号，以及明确的解决优化建议。 |
| **`TEST_`** | **Tests** (测试套件与验证用例) | 单元测试用例全覆盖、Headless 自动化试玩指令、自动化测试框架标准、测试补充专案。 | 必须详述测试用例的设计边界、断言逻辑、自动化脚本运行方式，以及测试覆盖率目标。 |
| **`HIST_`** | **Histories** (历史编年史与归档) | 开发日志大编年史、README 演进历史、Git 提交流水账、legacy 历史备忘录。 | 主要作为静态归档。要求内容充实，记录关键节点的实施大事件、版本冲突合并日志等。 |
| **`EXEC_`** | **Executions** (活动执行与交付物) | 当前开发会话计划书、待办任务检查清单、阶段性交付验证 Walkthrough 汇报。 | 作为开发执行过程的伴生文档。在会话开始时创建，在任务交付后完成归档记录。 |
| **`DICT_`** | **Dictionaries** (术语与命名词典) | 全局术语表、代码变量命名规范映射表、中英文术语对照字典。 | 必须包含规范的术语、代码映射、定义、首次出现文档与同义词黑名单。 |

### 3.2 [V1.2.0 新增] WFLOW_ 前缀

| 大写前缀 | 英文分类 (Category) | 适用文档范围 | 编写要求与规范 |
| :--- | :--- | :--- | :--- |
| **`WFLOW_`** | **Workflows** (工作流与操作规程) | 开发工作循环定义、AI 协作模式、会话启动协议、发布标准、日常操作手册。 | 描述"如何工作"的操作规程，与 SPEC_（"是什么"）和 EXEC_（"做了什么"）区分。WFLOW_ 文档通常长期有效，不随每次会话更新。 |

> [!NOTE]
> **SPEC_ vs WFLOW_ 区分标准**：SPEC_ 回答"系统应该怎么设计"（产品/技术规格），WFLOW_ 回答"我们应该怎么工作"（流程/操作规程）。例如：结局系统的触发条件设计是 SPEC_，AI 协作的日常开发循环是 WFLOW_。

### 3.3 [V1.2.0 新增] 活文档（Living Documents）

以下两个特殊文件位于代码仓库内（如 `src/test/scenarios/`），不遵循"新版本建新文件"的规则，持续原地更新：

| 文件名 | 用途 | 更新频率 |
| :--- | :--- | :--- |
| `_registry.md` | 场景测试注册表，所有可测试质量条目的 RED/GREEN 状态 | 每次修复后立即更新 |
| `_health.md` | 项目健康仪表盘，不可测试的风险与技术债追踪 | 每周或里程碑时更新 |

> [!IMPORTANT]
> 活文档是例外，不是常规。除上述两个文件外，其他所有文档仍遵循"新版本建新文件，保留旧版本"的原则。

## 📝 四、AI 协同与会话归档管理要求

### 4.1 EXEC_ 三位一体（保留，提高触发门槛）

当 AI 助手深度参与协同开发时，**重大功能迭代**必须遵循"三位一体"的执行归档规范：

```
[会话启动] ─> 创建 EXEC_[DATE]_PLAN.md (实施方案书)
              │
              └─> 创建 EXEC_[DATE]_TASK.md (任务进度清单)
                  │
[开发验证] ───────> 生成 EXEC_[DATE]_WALKTHROUGH.md (交付验证汇报) ─> [同步并推送至 GitHub]
```

### 4.2 [V1.2.0 新增] 触发门槛

| 工作类型 | 是否生成 EXEC_ 三位一体 | 替代方式 |
| :--- | :--- | :--- |
| **Bug 修复**（影响 ≤5 个文件） | ❌ 不生成 | 更新 `_registry.md`（RED→GREEN），Git commit 即可 |
| **功能迭代**（影响 >5 个文件） | ✅ 生成 | — |
| **重大架构调整** | ✅ 生成 + 额外生成 AUDIT_ | — |
| **发布前验收** | ✅ 生成 EXEC_WALKTHROUGH | 附带 registry 全绿截图 |

> [!TIP]
> **核心原则：能用测试表达的，不写文档。** Bug 修复的"交接物"是一个从 RED 变 GREEN 的测试，不是一份 WALKTHROUGH 文档。

### 4.3 EXEC_ 各文档要求（保持不变）

#### 1. 实施方案书 (`EXEC_[DATE]_PLAN.md`)
*   **目的**：说明在不修改底层数值/核心公式的原则下，准备如何进行重构或修复。
*   **内容**：说明设计痛点、要修改/新增的文件映射表、AI 执行约束。

#### 2. 任务进度清单 (`EXEC_[DATE]_TASK.md`)
*   **格式**：统一采用 markdown 的 TODO task 列表，如 `[ ]` 未启动、`[/]` 进行中、`[x]` 已完成。
*   **状态同步**：开发过程中，AI 必须随着进度的推进实时更新该文件的 TODO 状态。

#### 3. 交付验证汇报 (`EXEC_[DATE]_WALKTHROUGH.md`)
*   **目的**：向开发者证明代码的健壮性。
*   **硬性指标**：
     *   **测试运行记录**：列出单元测试或集成测试的运行通过率。
     *   **Registry 状态变化**：列出本次会话中哪些条目从 RED 变为 GREEN。
     *   **GitHub 同步状态**：详细列出推送目标分支及 remote 状态。

## 🎨 五、通用文档编写与视觉规范 (Best Practices)

为保持所有文档的高端品质和一致的阅读体验，编写新文档时应严格遵守以下格式要素：

1. **结构化头部信息 (Front Matter)**
   每个文档的顶部必须包含简明扼要的标头，便于读取：
   ```markdown
   # 文档标题
   > **版本号**: V1.0.0
   > **生效日期**: 2026-06-26
   > **基于版本**: [上一版本链接]（如适用）
   ```

2. **活用 GitHub 样式的高级警报 (Alerts)**
   使用通知框高亮关键信息，严禁连续使用或嵌套：
   *   `> [!NOTE]`：用于提供背景上下文或解释。
   *   `> [!TIP]`：用于性能优化建议、最佳实践或提效方案。
   *   `> [!IMPORTANT]`：用于必要要求、AI 关键步骤或强制约束。
   *   `> [!WARNING]`：用于防踩坑、不兼容更改或潜在越界问题。

3. **精准的文件链接与行号锚定**
   引用代码时，严禁使用模糊的描述。必须使用绝对文件链接，并精准链接到代码的行范围。

4. **主注册表目录同步**
   每个项目应在本地维护一个当前目录下所有文档的主索引文件。每当 AI 助手或开发者创建了新文档时，必须第一时间将新文档的文件链接与简要描述注册到该主文件中。

## 📋 六、[V1.2.0 新增] 文档 vs 测试决策规则

每次想"我应该写一份文档"时，先问：

```
这个问题能写成一个会失败的测试吗？
  │
  ├─ 能 → 写测试，加入 _registry.md，不写文档
  │
  └─ 不能 → 根据内容类型选择前缀，写文档
```

| 情境 | 正确做法 | 错误做法 |
| :--- | :--- | :--- |
| "人物解锁后UI不刷新" | 写 RED 场景测试 | 写 AUDIT_BUG_REPORT |
| "发现了12个bug" | 写 12 个 RED 测试 | 写 AUDIT_12_BUGS |
| "新纪元的触发逻辑设计" | 写 SPEC_ + 对应的 Design Drift 测试 | 只写 SPEC_ |
| "这次重构改了哪些文件" | 写 EXEC_WALKTHROUGH | — |
| "Game.ts 太长了需要拆分" | 记入 _health.md | 写 AUDIT_ARCHITECTURE |
